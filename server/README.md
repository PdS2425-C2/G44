## WebSocket

### Inviti a gruppi
**Descrizione del flusso logico:**

Un invito viene creato tramite una richiesta HTTP POST all'endpoint `/api/groups/{group_id}/requests`, non è necessario usare WebSocket in quanto il client contatta il server quando vuole creare l'invito.
Per permettere al client invitato una ricezione dell'invito in modo asincrono (quindi senza dover ricaricare la pagina o fare polling), il server, alla creazione della entry nella tabella `request`, invia un messaggio WebSocket a tutte le sessioni connesse dell'utente invitato. In questo caso è necessario utilizzare WebSocket perché il server deve essere in grado di inviare messaggi al client in modo asincrono, senza che il client debba fare una richiesta esplicita.

**Gestion WebSocket lato Rust:**
- Nell'`AppState` il campo `notification_peers` serve per associare ad ogni utente connesso (tramite il suo `user_id`) un insieme di canali di invio messaggi (`UnboundedSender<NotificationMessage>`). Ogni canale rappresenta una connessione WebSocket attiva per quell'utente. Nella maggior parte dei casi, un utente avrà un solo canale, ma se l'utente è connesso da più dispositivi o finestre del browser, ci saranno più canali associati al suo `user_id`. 
- Nella funzione `handle_notifications_socket`, viene creato un canale di comunicazione interno (usando `unbounded_channel`) per ogni connessione WebSocket. Il lato ricevente del canale (`rx`) è collegato al WebSocket, mentre il lato trasmittente (`tx`) viene memorizzato nell'`AppState` sotto l'`user_id` dell'utente. In questo modo, quando il server deve inviare una notifica a quell'utente, può utilizzare tutti i canali associati al suo `user_id` per inviare il messaggio a tutte le sue connessioni attive.
- Nella funzione `post_group_request`, dopo aver creato l'invito nella tabella `request`, il server costruisce un messaggio di notifica (`NotificationMessage::GroupRequestCreated`) e lo invia a tutti i canali associati all'`user_id` dell'utente invitato. In questo modo, tutte le connessioni WebSocket attive per quell'utente riceveranno la notifica dell'invito.
- Per il momento se un utente chiude una connessione WebSocket, il canale associato rimane nell'`AppState`, ma non causa problemi significativi. In futuro, si potrebbe implementare una logica di pulizia per rimuovere i canali non più attivi dall'`AppState` quando una connessione WebSocket viene chiusa. Un codice non funzionante per fare ciò è già presente, commentato, in fondo alla funzione `handle_notifications_socket`.
